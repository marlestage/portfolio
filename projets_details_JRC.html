Jeu de Rapidité Collaboratif (JRC)

Système embarqué temps réel & protocole série

Projet en cours : Développement de l'interface multijoueur Python

Mission : Conception intégrale (Hardware & Software) d'un module de jeu connecté type jeu de taupe. En tant que "responsable" logicielle, j'ai assuré le développement d'un firmware temps réel performant et d'un protocole de communication robuste avec un PC superviseur.

Objectif final : Interconnecter plusieurs manettes JRC via une interface Python pour organiser des tournois interactifs.

1
Analyse (FAD) : Choix des composants (ATmega328P, LEDs RGB) via matrice de décision.
2
Hardware (DDF) : Routage du PCB et assemblage des composants (CMS/THT).
3
Software : Machine à états, protocole série et gestion EEPROM.
4
Validation (DDV) : Tests fonctionnels et identifiant unique.

Module joueur assemblé

1. Conception système & Choix matériels

Le projet imposait des contraintes strictes de coût et de performance. J'ai réalisé des Fiches d'Aide à la Décision (FAD) pour justifier chaque choix :

  • Microcontrôleur : Choix de l'ATmega328P (Arduino Nano) pour son coût réduit (2,29€) et sa facilité de programmation en C++, tout en offrant suffisamment de broches PWM pour les LEDs RGB.
  • Interface : Boutons poussoirs MCDTS6-5R pour leur réactivité et la facilité d'appui. LEDs RGB ASMB-MTB0 pour l'affichage des états de jeu dans les différentes couleurs demandées.
  • Communication : Conversion USB-UART pour dialoguer avec le PC maître.

Synoptique Système (Hardware)

2. Architecture logicielle

Machine à états & Protocole Série

J'ai développé l'intégralité du code en C++. L'architecture repose sur une machine à états non-bloquante (pas de delay()) pour gérer simultanément l'attente d'ordres, la lecture des boutons et l'affichage LED.

J'ai implémenté un protocole de communication robuste pour décoder les trames reçues :

  • A : Lance le jeu A (réflexe simple).
  • B : Lance le jeu B (variante à 20 appuis consécutifs).
  • R : Reset du module (remise à zéro).
  • V + param : Demande d'infos (0=ID, 1=Dernier temps, 2=Erreurs, 3=Valides, 4=Moyenne).

Identification unique (EEPROM)

Pour permettre le jeu multijoueur, chaque manette possède un identifiant unique stocké en mémoire non-volatile (EEPROM).
Notre module porte l'identifiant C111 (Classe C1, Groupe 1, Prototype 1), ce qui permettra au futur logiciel Python de distinguer notre manette des autres lors des tournois.

// Extrait : Traitement de la trame série
void traiterTrame(String trame){
  char id = trame[0];
  if(id == 'R') { resetModule(); }
  else if(id == 'V') { envoyerInformation(trame[1]); }
  else if(id == 'A'){ jeuA(trame[1], trame[2], trame[3]); }
  else if(id == 'B'){ jeuB(trame[1], trame[2]); }
}

3. Fabrication & Vérification

Fabrication (DDF) : Nous avons réalisé l'assemblage de la carte en utilisant la technique de refusion pour les composants CMS et le soudage manuel pour les traversants.

Validation (DDV) : J'ai piloté les tests unitaires pour vérifier la conformité aux exigences :

  • Dimensions mécaniques et fixations (Non-conformité mineure relevée et corrigée).
  • Consommation énergétique via USB (< 500mA).
  • Réactivité des algorithmes de jeu A et B (Validé).
